<objective>
Turn ideas into approved designs through dialogue before any implementation action.
</objective>

<essential_principles>
<HARD-GATE>
No implementation action — code, scaffolding, project creation, or skill invocation — until a design has been presented AND user has approved it. This applies to EVERY project regardless of perceived simplicity.
</HARD-GATE>

- **One question at a time.** Don't overwhelm.
- **Multiple choice preferred** over open-ended when possible.
- **YAGNI ruthlessly** — remove unnecessary from designs.
- **Explore alternatives** — always propose 2-3 approaches before settling.
- **Design isolation** — each component has one clear purpose with well-defined interfaces.
</essential_principles>

<intake>
Ask what the user wants to design or build. Explore purpose, constraints, and success criteria through focused questions.
</intake>

<routing>
Route to workflows/design-process.md for the full design checklist and process flow.
</routing>

<quick_reference>
- Explore project context first
- Ask clarifying questions one at a time
- Propose 2-3 approaches with trade-offs
- Present design and get approval
- Write spec to docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md
- Self-review spec for placeholders, consistency, scope, ambiguity
- User reviews and approves spec
- Invoke writing-plans skill for implementation
</quick_reference>
